# Review Steering - ステアリングファイルレビュー

このスキルは、`/plan-task` で作成されたステアリングファイルを実装開始前にレビューし、品質と実装準備状態を検証します。

## Parameters

- `$1`: ステアリングディレクトリ名（例: `20250409-add-backend-phase1`）

## 使用方法

```
/review-steering 20250409-add-backend-phase1
```

**利用可能なステアリングディレクトリを確認:**
```bash
ls -1 .steering/
```

---

## 実行手順

### 1. 引数の確認

引数 `` が提供されているか確認してください。

**引数なしの場合:**
```
エラー: ステアリングディレクトリ名を指定してください。

使用方法: /review-steering <steering-directory-name>

利用可能なステアリングディレクトリ:
```
```bash
ls -1 .steering/
```

### 2. ステアリングファイルの読み込み

以下のファイルをすべて読み込んでください：
```
.steering/$1/requirements.md
.steering/$1/design.md
.steering/$1/tasklist.md
```

### 3. 関連ドキュメントの読み込み

以下のドキュメントも読み込んでください：
```
docs/glossary.md
docs/system-architecture.md
docs/security-guidelines.md
docs/jsox-compliance.md
docs/ENVIRONMENT.md
docs/service-contracts.md
contracts/openapi/bff-api.yaml
contracts/proto/*.proto（存在する場合）
```

前提タスク（依存ステアリング）がある場合はそのファイルも読み込む。

### 4. レビュー実行

以下の7カテゴリのチェックを順次実行してください。

---

## レビュー項目（7カテゴリ）

### 1. 構造・完全性チェック

#### ファイル存在確認
- [ ] `requirements.md` が存在するか
- [ ] `design.md` が存在するか
- [ ] `tasklist.md` が存在するか

#### requirements.md 必須セクション
- [ ] 目的が記載されているか
- [ ] スコープ（実装するもの）が明記されているか
- [ ] スコープ外（実装しないもの）が明記されているか
- [ ] ユーザーストーリーまたは受け入れ条件があるか
- [ ] 制約事項が記載されているか
- [ ] 成功の定義があるか
- [ ] 前提タスク（依存ステアリング）が明記されているか（ある場合）

#### design.md 必須セクション
- [ ] アーキテクチャ図（Mermaid等）があるか
- [ ] API設計（エンドポイント/RPC定義）が具体的か
- [ ] データベース設計（テーブル定義、SQL）があるか（DB変更がある場合）
- [ ] コンポーネント構成が記載されているか
- [ ] エラーハンドリング方針が記載されているか

#### tasklist.md 必須セクション
- [ ] Agent別タスク分担が明記されているか
- [ ] 依存関係が図示されているか
- [ ] 実装順序（推奨）が定義されているか
- [ ] 完了条件が各Agentごとに定義されているか

---

### 2. 3ファイル間の整合性

- [ ] requirements.mdの「実装するもの」がすべてdesign.mdに設計されているか
- [ ] design.mdのコンポーネント/API/テーブルがtasklist.mdにタスクとして分解されているか
- [ ] requirements.mdの受け入れ条件がtasklist.mdの完了条件に反映されているか
- [ ] requirements.mdの制約事項がdesign.mdの設計判断に反映されているか
- [ ] 用語が3ファイル間で統一されているか

**チェック方法:**
1. requirements.mdの「実装するもの」を箇条書きで抽出
2. 各項目がdesign.mdに対応する設計記述があるか確認
3. 各項目がtasklist.mdに対応するタスクがあるか確認
4. 漏れがあればレポートに記載

---

### 3. 既存ドキュメントとの整合性

#### 用語統一（glossary.md）
- [ ] ステアリングファイル内で使用されている用語がglossary.mdに定義されているか
- [ ] 新規用語がある場合、glossary.md更新の必要性が言及されているか
- [ ] コード上の命名規則（英語名、変数名）がglossary.mdに準拠しているか

#### アーキテクチャ整合性（system-architecture.md）
- [ ] design.mdのアーキテクチャがsystem-architecture.mdと矛盾しないか
- [ ] サービス間通信方式（REST/gRPC）が一致しているか
- [ ] ポート番号がsystem-architecture.mdおよびENVIRONMENT.mdと一致しているか

#### セキュリティ要件（security-guidelines.md）
- [ ] 認証・認可要件が反映されているか
- [ ] データ保護（暗号化、ハッシュ化）要件が反映されているか

#### J-SOX要件（jsox-compliance.md）
- [ ] 監査ログ要件が反映されているか
- [ ] 変更履歴記録が必要な場合、設計に含まれているか
- [ ] 職務分掌が必要な場合、設計に含まれているか

#### 環境設定（ENVIRONMENT.md）
- [ ] ポート番号がENVIRONMENT.mdと一致しているか
- [ ] 環境変数がENVIRONMENT.mdに記載されているか（または追記が必要か）

---

### 4. API契約・DB設計チェック

#### API契約
- [ ] 新規APIがdesign.mdに十分な詳細（リクエスト/レスポンス型、ステータスコード）で定義されているか
- [ ] 既存の `contracts/openapi/` や `contracts/proto/` と矛盾しないか
- [ ] 後方互換性が維持されるか（既存APIの変更がある場合）
- [ ] API契約の更新ファイル（OpenAPI/Proto）が明示されているか

#### DB設計
- [ ] テーブル定義にPK、FK、インデックス、制約が定義されているか
- [ ] 既存テーブルとの関連（FK等）が正しいか
- [ ] マイグレーションファイルの命名規則（V1__, V2__...）が既存と衝突しないか
- [ ] 初期データ（seed）が適切に定義されているか

---

### 5. タスクリスト品質

#### Agent分担
- [ ] 各Agentの担当範囲が明確か
- [ ] Agent間の境界（どのファイルを誰が触るか）が明確か
- [ ] 単一Agentで完結するタスクはAgent Teamsを使わず通常のClaude Codeで実行すべきか検討

#### 依存関係
- [ ] Agent間の依存関係が定義されているか
- [ ] クリティカルパス（最長パス）が特定されているか
- [ ] 並行実行可能なタスクが明示されているか
- [ ] 循環依存がないか

#### タスク粒度
- [ ] 各タスクが1つの明確な成果物に対応しているか
- [ ] タスクが大きすぎないか（1タスクでファイル5個以上変更は要分割）
- [ ] タスクが小さすぎないか（意味のある単位か）

#### 完了条件
- [ ] 各Agentの完了条件が具体的か（「テストが通る」ではなく「何のテストが通る」）
- [ ] 完了条件が検証可能か（自動テスト、コマンド実行等）

---

### 6. リスク・実現可能性

#### スコープサイズ
- [ ] 影響するファイル数の見積もりが適切か
- [ ] タスク分割の判定基準（30ファイル以上、7日以上等）に照らして妥当か
- [ ] 分割すべきなのに分割されていないケースがないか

#### 技術的リスク
- [ ] 未知の技術・ライブラリを使用する箇所が特定されているか
- [ ] 外部サービスへの依存がある場合、フォールバックが検討されているか
- [ ] パフォーマンス要件が実現可能か

#### 設計仮説の実測検証 (2026-04-14 振り返り由来)
- [ ] design.md の「〜できない」「〜が必要」の断定が実測で検証されているか
- [ ] 検証されていない断定がある場合、「仮説」であることが明示されているか
- [ ] 5 分以内に検証できる断定 (`cat` / `grep` / `curl -I` / `\d table` 等) が未検証のまま残っていないか
- [ ] 既存コードパターンを「表面的観察」ではなく「意図を読み解いた理解」で参照しているか

**検証例:**
- 「BFF session Cookie は browser に転用不可」→ `cat .auth/*.json` で Domain 属性を確認
- 「既存 X spec はパターン Y のため」→ git blame でコミットメッセージと経緯を確認
- 「ライブラリ Z は機能 W 未サポート」→ docs / changelog を 1 件 grep
- 「DB table T に制約 C がある」→ `psql -c '\d T'`

検証できない断定があっても即 Blocker にはならないが、review-report.md で **要検証** として記録し、
実装時に再確認することを明示する。

#### 環境設定
- [ ] Docker Compose設定がdesign.mdに含まれているか
- [ ] ポート番号の衝突がないか（既存サービスとの共存）
- [ ] 環境変数が網羅されているか

---

### 7. Agent Teams実装準備状態チェック

**このカテゴリはAgent Teamsで実装を開始できる状態かを判定します。**

#### 前提条件の充足
- [ ] 前提タスク（依存ステアリング）の実装が完了しているか
- [ ] 前提タスクの成果物（コード、API契約、DB等）が利用可能な状態か
- [ ] サブモジュールが最新状態か（`git submodule update --init --recursive`）

#### API契約の確定
- [ ] API契約ファイル（OpenAPI/Proto）が `contracts/` に配置済みか、またはステアリング内で定義されているか
- [ ] 複数Agentが参照するAPI契約が事前に確定しているか
- [ ] API契約の未確定部分がある場合、どのAgentが確定するか明記されているか

#### 各Agentの自律実行可能性
- [ ] 各Agentがステアリングファイルだけで作業内容を理解できるか
- [ ] 各Agentが参照すべきドキュメント一覧が明記されているか
- [ ] 各Agentが他Agentの完了を待たずに開始できる部分があるか（並行度）
- [ ] Agentへの指示に曖昧な部分がないか（「適切に実装」等の曖昧表現）

#### 環境・ツールの準備
- [ ] 必要な開発ツール（protoc, sqlc, npm等）のインストール手順が明確か
- [ ] Docker Compose設定が定義されているか（DB、マイグレーション等）
- [ ] テスト実行方法が明記されているか
- [ ] コミット・プッシュのブランチ戦略が明確か

#### サービス別CLAUDE.mdとの整合
- [ ] 対象サービスのCLAUDE.mdが存在し、技術スタック・規約が定義されているか
- [ ] CLAUDE.mdの開発規約とステアリングの設計が矛盾しないか
- [ ] CLAUDE.mdに記載のないライブラリ・技術を使用していないか（使用する場合はCLAUDE.md更新が必要）

#### Agent Teams起動判定

以下の**すべて**を満たす場合、Agent Teams実装開始可能と判定：

| # | 条件 | 必須 |
|---|------|------|
| 1 | 3ファイル（requirements/design/tasklist）が揃っている | ✅ |
| 2 | 3ファイル間の整合性に重大な問題がない | ✅ |
| 3 | API契約が確定している（または確定手順が明記） | ✅ |
| 4 | 前提タスクが完了している | ✅ |
| 5 | 各Agentの担当範囲・完了条件が明確 | ✅ |
| 6 | 対象サービスのCLAUDE.mdが存在する | ✅ |
| 7 | 環境設定（ポート、DB、Docker）が定義されている | ✅ |

**判定結果:**
- **✅ Ready**: すべて満たす → `/start-implementation` で実装開始可能
- **⚠️ Conditional**: 軽微な問題あり → 問題を修正してから実装開始
- **❌ Not Ready**: 重大な問題あり → ステアリングファイルの修正が必要

## 注意事項

- レビュー結果は `.steering/$1/steering-review.md` として保存します
- ❌（要修正）がある場合は、修正後に再度レビューを実行してください
- Agent Teams準備状態が ❌ Not Ready の場合は `/start-implementation` を実行しないでください
- レビュー後にステアリングファイルを修正した場合は、mainブランチにコミット・プッシュしてください
