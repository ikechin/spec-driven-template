# Analyze Impact - 依存関係影響分析

このスキルは、変更が他サービスに与える影響を分析し、必要な対応を明確にします。

## Parameters

- `$1`: 変更内容の説明（オプション。指定しない場合は対話形式でヒアリング）

## 使用方法

```
/analyze-impact "BFF APIに新しいエンドポイントを追加"
```

または

```
/analyze-impact
```

---

## 分析フロー

### 1. 変更内容のヒアリング

以下の情報を収集します：

- **変更の種類**: 新規機能追加 / 既存機能修正 / リファクタリング / 削除
- **対象サービス**: Frontend / BFF / Backend / E2E
- **変更内容の詳細**: 具体的に何を変更するか

### 2. 影響範囲の分析

#### 2.1 API契約の影響

```markdown
### API契約変更チェック

- [ ] **新規エンドポイント追加**
  - 影響: Frontend（新しいAPI呼び出し実装が必要）
  - 対応: `contracts/openapi/` に仕様追加
  - 後方互換性: 影響なし

- [ ] **既存エンドポイント修正**
  - リクエスト/レスポンスの型変更
  - HTTPステータスコード変更
  - 影響: Frontend（API呼び出し側の修正が必要）
  - 対応: バージョニング or 後方互換性維持
  - 後方互換性: 破壊的変更の有無を確認

- [ ] **エンドポイント削除**
  - 影響: Frontend（代替APIへの移行が必要）
  - 対応: 非推奨期間を設ける
  - 後方互換性: 破壊的変更
```

#### 2.2 データベース変更の影響

```markdown
### DB変更チェック

- [ ] **テーブル追加**
  - 影響: Backend（新しいCRUD操作）, BFF（新しいAPI）
  - 対応: Flywayマイグレーション作成
  - ロールバック: マイグレーションのダウンスクリプト必要

- [ ] **カラム追加**
  - 影響: Backend（モデル修正）
  - 対応: デフォルト値の設定
  - 後方互換性: NULL許可 or デフォルト値で維持

- [ ] **カラム変更・削除**
  - 影響: Backend（既存データの移行が必要）
  - 対応: データ移行スクリプト
  - 後方互換性: 破壊的変更の可能性

- [ ] **インデックス追加**
  - 影響: パフォーマンス向上
  - 対応: 本番DBでのインデックス作成時間を考慮
```

#### 2.3 共通型・ユビキタス言語の影響

```markdown
### 共通型変更チェック

- [ ] **contracts/types/ の型変更**
  - 影響: Frontend, BFF, Backend（型定義の更新が必要）
  - 対応: すべてのサービスで型を再生成

- [ ] **docs/glossary.md の用語変更**
  - 影響: すべてのサービス（コード上の命名変更）
  - 対応: リファクタリング（変数名、関数名、型名）
  - 影響範囲: grepで使用箇所を検索

- [ ] **新規用語追加**
  - 影響: 該当機能を実装するサービスのみ
  - 対応: `docs/glossary.md` に追加
```

#### 2.4 認証・認可の影響

```markdown
### 認証・認可変更チェック

- [ ] **新しい権限追加**
  - 影響: Backend（権限定義）, BFF（権限チェック）, Frontend（UI表示制御）
  - 対応: 権限マスタの更新、ミドルウェアの修正

- [ ] **認証方式変更**
  - 影響: すべてのサービス
  - 対応: セッション管理、トークン管理の見直し
  - 後方互換性: 破壊的変更
```

#### 2.5 環境変数・設定の影響

```markdown
### 環境変数変更チェック

- [ ] **新しい環境変数追加**
  - 影響: 該当サービスの .env ファイル
  - 対応: `docs/ENVIRONMENT.md` に追加
  - デプロイ時: 本番環境の環境変数設定が必要

- [ ] **環境変数削除**
  - 影響: 使用していたサービス
  - 対応: コードから参照を削除
```

### 3. マイグレーション手順の策定

変更を安全に適用するための手順を策定します：

```markdown
## マイグレーション手順

### ステップ1: Backend
1. DBマイグレーション実行
2. モデル・ドメインロジック修正
3. ユニットテスト実行

### ステップ2: BFF
1. Backend APIの変更を確認
2. BFF APIの実装
3. OpenAPI仕様更新 (`contracts/openapi/`)
4. ユニットテスト実行

### ステップ3: Frontend
1. OpenAPI仕様から型生成
2. 新しいAPIを呼び出すコード実装
3. UIテスト実行

### ステップ4: E2E
1. 統合E2Eテスト実行
2. 全サービス連携確認
```

### 4. 後方互換性の確認

```markdown
## 後方互換性チェックリスト

- [ ] **既存のAPIエンドポイントが引き続き動作するか**
- [ ] **既存のデータが正しく扱えるか**
- [ ] **古いクライアントが引き続き動作するか**
- [ ] **ロールバック可能か**

### 破壊的変更がある場合の対応
1. バージョニング（/api/v2/ を新設）
2. 非推奨期間を設ける（3ヶ月等）
3. クライアントへの移行通知
4. 段階的なロールアウト
```

### 5. 必要な更新ドキュメントのリスト化

```markdown
## 更新が必要なドキュメント

### 永続的ドキュメント（docs/）
- [ ] `docs/glossary.md` - 新規用語の追加
- [ ] `docs/service-contracts.md` - API契約の更新
- [ ] `docs/system-architecture.md` - アーキテクチャ変更（該当する場合）
- [ ] `docs/ENVIRONMENT.md` - 環境変数の追加・変更

### サービス別ドキュメント
- [ ] `services/frontend/docs/functional-design.md`
- [ ] `services/bff/docs/functional-design.md`
- [ ] `services/backend/docs/functional-design.md`

### API仕様
- [ ] `contracts/openapi/bff-api.yaml`

### ステアリングファイル
- [ ] `.steering/[YYYYMMDD]-[タスク名]/design.md` - 影響分析結果を記録
```

## 出力形式

```markdown
# Impact Analysis Report

## 変更内容
- **変更の種類**: 既存機能修正
- **対象サービス**: BFF, Backend
- **変更内容**: 加盟店検索にタグフィルタ機能を追加

## 影響範囲サマリー

| サービス | 影響度 | 変更内容 | 破壊的変更 |
|---------|--------|---------|-----------|
| Frontend | 高 | 新しいAPIパラメータ追加 | なし |
| BFF | 高 | エンドポイント修正 | なし |
| Backend | 高 | 検索ロジック追加 | なし |
| E2E | 中 | 新しいテストケース追加 | なし |

## 詳細分析

### 1. API契約の影響
**変更内容:**
- GET `/api/v1/merchants` に `tags` クエリパラメータを追加

**影響:**
- Frontend: 新しいパラメータを送信するコード追加
- BFF: クエリパラメータの受け取り・バリデーション
- Backend: タグフィルタロジック実装

**後方互換性:** ✅ 維持される（オプショナルパラメータのため）

### 2. データベース変更の影響
**変更内容:**
- `merchants` テーブルに `tags` カラム（TEXT[]）を追加

**影響:**
- Backend: モデルに `tags` フィールド追加
- マイグレーション: Flyway V003__add_tags_to_merchants.sql

**後方互換性:** ✅ 維持される（NULL許可）

### 3. 共通型の影響
**変更内容:**
- `Merchant` 型に `tags: string[]` を追加

**影響:**
- Frontend: 型再生成が必要
- BFF: 型再生成が必要

### 4. 認証・認可の影響
影響なし

### 5. 環境変数の影響
影響なし

## マイグレーション手順

### ステップ1: Backend（優先度: 最高）
1. DBマイグレーション作成・実行
2. Merchant モデルに tags フィールド追加
3. タグフィルタロジック実装
4. ユニットテスト追加

### ステップ2: BFF（優先度: 高）
1. OpenAPI仕様更新（tags パラメータ追加）
2. クエリパラメータのバリデーション追加
3. Backend API呼び出しに tags を渡す
4. ユニットテスト追加

### ステップ3: Frontend（優先度: 中）
1. OpenAPI から型再生成
2. タグフィルタUIコンポーネント追加
3. API呼び出しに tags パラメータ追加
4. UIテスト追加

### ステップ4: E2E（優先度: 中）
1. タグフィルタのE2Eテスト追加

## 後方互換性
✅ **後方互換性は維持されます**

理由:
- `tags` パラメータはオプショナル
- 既存のAPIリクエストは引き続き動作
- DBカラムはNULL許可

## 更新が必要なドキュメント

- [ ] `contracts/openapi/bff-api.yaml` - tags パラメータ追加
- [ ] `docs/glossary.md` - "タグ" の定義追加（該当する場合）
- [ ] `.steering/[YYYYMMDD]-add-tag-feature/design.md` - 影響分析結果を記録

## リスク評価

| リスク | 確率 | 影響度 | 対策 |
|-------|------|--------|------|
| DBマイグレーション失敗 | 低 | 高 | ロールバックスクリプト準備 |
| パフォーマンス劣化 | 中 | 中 | tags カラムにインデックス追加 |
| 既存機能への影響 | 低 | 中 | 十分なE2Eテスト |

## 推奨事項
1. tags カラムに GIN インデックスを追加してパフォーマンス向上
2. タグの値を正規化（小文字統一等）
3. タグ数の上限を設定（例: 10個まで）
```

---

## 注意事項

- 破壊的変更がある場合は、必ずバージョニングや非推奨期間を検討してください
- マイクロサービス間の依存関係を考慮し、適切な順序で変更を適用してください
- 本番環境への適用前に、ステージング環境で十分にテストしてください
- 影響分析結果は、該当するステアリングファイルに記録してください
