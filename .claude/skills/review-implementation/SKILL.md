# Review Implementation - 成果物レビュー

このスキルは、Agent Teamsで実装された成果物がステアリングファイルの仕様に準拠しているかを包括的にレビューします。

## ⚠️ レビュアー向けガードレール (2026-04-14 振り返り由来)

レビュー対象のテスト/静的解析が **すべてクリーン** な場合、以下の種類の指摘は **誤検出の可能性が高い** ため Critical として報告する前に必ず二次確認する:

1. **シンボル未定義エラー** — `go vet` / `go test` / `tsc` が通っているなら、定義はどこかに存在する (別ファイル、同一パッケージ、import 先等)。言語のスコープルールを再確認してから報告すること
2. **型不一致** — 型チェックが通っているなら実行時不整合はない
3. **import 漏れ** — コンパイルが通っているなら import は足りている

誤検出の具体例 (`20260414-approval-history-search`): Go の同一 package 内 const `defaultPageSize` が別ファイル (`merchant_service.go`) で定義されていたのを「未定義」と誤報告。Go package scope の基本を見落としていた。

**ルール**: 静的解析/テストがクリーンな場合、言語仕様レベルの違反指摘は **必ず Grep で当該シンボルの定義箇所を確認** してから報告する。見つからない場合のみ Critical 候補とする。

## Parameters

- `$1`: ステアリングディレクトリ名（例: `20250407-frontend-bff-only`）

## 使用方法

```
/review-implementation 20250407-frontend-bff-only
```

## レビュー項目

### 1. ステアリングファイル準拠性チェック

#### 1.1 requirements.md との整合性
- [ ] 受け入れ条件がすべて満たされているか
- [ ] ユーザーストーリーが実装されているか
- [ ] 制約事項が遵守されているか
- [ ] 影響範囲（対象サービス）が正しいか

#### 1.2 design.md との整合性
- [ ] 設計通りのアーキテクチャで実装されているか
- [ ] データ構造が設計通りか
- [ ] コンポーネント構成が設計通りか
- [ ] 実装アプローチが設計に従っているか

#### 1.3 tasklist.md との整合性
- [ ] すべてのタスクが完了しているか
- [ ] Agent別の担当範囲が正しく実装されているか
- [ ] タスク間の依存関係が解決されているか

### 2. 機能過不足チェック

#### 2.1 機能過剰チェック
- [ ] 仕様にない余計な機能を追加していないか
- [ ] スコープ外の実装がないか
- [ ] 過度な抽象化・汎用化をしていないか

#### 2.2 機能不足チェック
- [ ] 必須機能がすべて実装されているか
- [ ] エッジケースの処理が漏れていないか
- [ ] エラーハンドリングが適切か

### 3. API契約準拠性チェック

- [ ] `contracts/openapi/` の仕様と実装が一致しているか
- [ ] リクエスト・レスポンスの型が正しいか
- [ ] HTTPステータスコードが仕様通りか
- [ ] エラーレスポンスの形式が統一されているか
- [ ] バージョニングが適切か

### 4. 用語統一チェック

- [ ] `docs/glossary.md` のユビキタス言語が使われているか
- [ ] 変数名・関数名・型名が用語集に準拠しているか
- [ ] コメント・ログメッセージで用語が統一されているか
- [ ] 新規用語が追加された場合、glossary.mdが更新されているか

### 5. セキュリティチェック

#### 5.1 認証・認可
- [ ] `docs/security-guidelines.md` の認証方式に準拠しているか
- [ ] 認可チェックが適切に実装されているか（権限チェック）
- [ ] セッション管理が安全か（HttpOnly Cookie等）
- [ ] CSRF対策が実装されているか（該当する場合）

#### 5.2 入力検証
- [ ] すべてのユーザー入力がバリデーションされているか
- [ ] SQLインジェクション対策がされているか
- [ ] XSS対策がされているか（フロントエンド）
- [ ] パストラバーサル対策がされているか

#### 5.3 データ保護
- [ ] 機密情報（パスワード、トークン等）が適切に暗号化されているか
- [ ] パスワードがハッシュ化されているか（bcrypt等）
- [ ] 環境変数から機密情報を読み込んでいるか（ハードコード禁止）
- [ ] ログに機密情報が出力されていないか

#### 5.4 その他セキュリティ
- [ ] 依存ライブラリに既知の脆弱性がないか
- [ ] HTTPSが強制されているか（本番環境）
- [ ] セキュリティヘッダーが適切に設定されているか

### 6. パフォーマンスチェック

#### 6.1 データベースクエリ
- [ ] N+1問題が発生していないか
- [ ] 適切なインデックスが設定されているか
- [ ] 不要なカラムを取得していないか（SELECT *の乱用）
- [ ] ページネーションが実装されているか（大量データの場合）

#### 6.2 API・ネットワーク
- [ ] 不要なAPI呼び出しがないか
- [ ] レスポンスサイズが適切か
- [ ] キャッシュが適切に利用されているか
- [ ] タイムアウト設定が適切か

#### 6.3 フロントエンド
- [ ] 不要な再レンダリングがないか（React.memo等）
- [ ] 画像が最適化されているか
- [ ] バンドルサイズが適切か
- [ ] コード分割が適切か（動的import）

#### 6.4 その他パフォーマンス
- [ ] メモリリークがないか
- [ ] 無限ループ・再帰の制御があるか
- [ ] 重い処理が非同期化されているか

### 7. J-SOX準拠チェック

#### 7.1 監査証跡
- [ ] `docs/jsox-compliance.md` の監査ログ設計に準拠しているか
- [ ] すべての重要な操作がログに記録されているか
  - Who（誰が）: ユーザーID
  - When（いつ）: タイムスタンプ
  - What（何を）: 操作内容
  - Where（どこで）: IPアドレス、エンドポイント
  - Result（結果）: 成功/失敗
- [ ] ログが改ざん不可能な形で保存されているか

#### 7.2 職務分掌
- [ ] 権限が適切に分離されているか
- [ ] 承認フローが実装されているか（該当する場合）
- [ ] 兼務禁止の制御があるか（該当する場合）

#### 7.3 アクセス制御
- [ ] ロールベースアクセス制御（RBAC）が実装されているか
- [ ] 最小権限の原則が守られているか
- [ ] データアクセス権限が適切に制御されているか

#### 7.4 データ保護
- [ ] 個人情報・機密情報が適切に保護されているか
- [ ] データバックアップの仕組みがあるか
- [ ] データ削除時の監査ログが残るか

### 8. コード品質チェック

#### 8.1 静的解析
- [ ] リントエラーがないか（ESLint, golangci-lint等）
- [ ] 型チェックエラーがないか（TypeScript, Go）
- [ ] フォーマットが統一されているか（Prettier, gofmt）

#### 8.2 テスト
- [ ] 単体テストが書かれているか
- [ ] テストカバレッジが十分か（目安: 80%以上）
- [ ] E2Eテストが書かれているか（該当する場合）
- [ ] テストが成功しているか

#### 8.3 コードの可読性
- [ ] 適切な命名がされているか
- [ ] 複雑な処理にコメントがあるか
- [ ] 関数・メソッドが適切なサイズか（長すぎない）
- [ ] 重複コードがないか（DRY原則）

### 9. ドキュメント整合性チェック

#### 9.1 永続的ドキュメント
- [ ] `docs/glossary.md` が更新されているか（新規用語追加時）
- [ ] `docs/service-contracts.md` が更新されているか（API変更時）
- [ ] サービス別の `docs/` が更新されているか（アーキテクチャ変更時）

#### 9.2 ステアリングファイル
- [ ] `tasklist.md` のすべてのタスクが completed になっているか
- [ ] 実装中に判明した変更点が記録されているか

## レビュー実行フロー

### コンテキスト節約の原則

**全ファイルをReadで読み込むのではなく、`git diff` で差分を特定してからレビューする。**
これによりコンテキストウィンドウの消費を大幅に削減できる。

### 手順

#### ステップ1: ステアリングファイル読み込み（必須）

`.steering/$1/` 配下の3ファイルを読み込む。これはレビュー基準となるため全文読み込みが必要。

```
Read .steering/$1/requirements.md
Read .steering/$1/design.md
Read .steering/$1/tasklist.md
```

#### ステップ2: 対象サービスと差分の特定

requirements.md から対象サービスを特定し、**`git diff` で変更差分を取得する**。
全ファイルをReadするのではなく、差分ベースでレビューを行う。

**サブモジュールの場合（各サービスが独立リポジトリ）:**
```bash
# 対象サブモジュールに移動してfeatureブランチの差分を取得
cd services/{service}
git diff main...HEAD          # mainからの全変更差分
git diff main...HEAD --stat   # 変更ファイル一覧（概要把握用）
```

**親リポジトリの場合:**
```bash
git diff main...HEAD
git diff main...HEAD --stat
```

**差分が大きい場合の段階的アプローチ:**
```bash
# 1. まず変更ファイル一覧で全体像を把握
git diff main...HEAD --stat

# 2. ファイル種別ごとに差分を確認（コンテキスト節約）
git diff main...HEAD -- '*.go' --no-ext-diff     # Goソースのみ
git diff main...HEAD -- '*.sql'                    # SQLのみ
git diff main...HEAD -- '*.yaml' '*.yml'           # 設定ファイルのみ
git diff main...HEAD -- 'Dockerfile' 'Makefile'    # ビルド関連のみ

# 3. 特定ファイルを深堀りする必要がある場合のみReadを使用
#    （差分だけでは前後のコンテキストが不足する場合）
```

#### ステップ3: 静的解析・テスト実行

差分の読み込みと並行して、自動チェックを実行する。

```bash
# Go の場合
go test ./... -v -count=1     # テスト実行
go vet ./...                   # 静的解析
go fmt ./...                   # フォーマットチェック（差分があれば問題）

# Frontend (Next.js/React) の場合
npm run type-check             # 型チェック
npm run lint                   # ESLint
npm run test                   # テスト

# BFF (Go + Echo) の場合
go test ./... -v -count=1
go vet ./...
```

#### ステップ4: 差分ベースでチェックリスト実行

`git diff` の出力をもとに、上記9カテゴリのレビュー項目をチェックする。

**差分だけでは判断できない場合のみ**、`Read` で特定ファイルの特定行を読む：
```
Read file_path offset=100 limit=30   # 必要な箇所のみ、行範囲を絞って読む
```

#### ステップ5: 問題点のレポート

問題点を重要度別に分類して報告する：
- **Critical（重大）**: セキュリティ脆弱性、J-SOX違反、仕様の重大な逸脱
- **High（高）**: 機能不足、パフォーマンス問題、API契約違反
- **Medium（中）**: コード品質問題、用語不統一、ドキュメント未更新
- **Low（軽微）**: コードスタイル、コメント不足

#### ステップ6: 改善提案

問題点の修正方法を具体的に提案する。`git diff` の差分行番号を参照して場所を特定する。

## 注意事項

- レビューは自動化できる部分と人間の判断が必要な部分があります
- Critical/High問題は必ず修正してください
- Medium/Low問題は優先度に応じて対応してください
- レビュー結果を `.steering/$1/review-report.md` として保存します
