# Plan Task - タスク計画ガイド

このスキルは、新しいタスクのステアリングファイルを作成するために必要な情報を構造化してヒアリングします。

## ヒアリング項目

以下の項目について順番に質問し、回答を収集します：

### 1. タスクの基本情報
- **タスク名**: 簡潔な名前（例: add-tag-feature, fix-filter-bug）
- **概要**: 何を実装・修正するか（1-2文）
- **背景・目的**: なぜこのタスクが必要か

### 2. 影響範囲
- **対象サービス**: □ Frontend □ BFF □ Backend □ E2E
- **既存機能への影響**: 破壊的変更はあるか？
- **他のタスク・機能との依存関係**: あるか？

### 2.5 タスクサイズ判定

以下の質問でタスクの規模を判定します：

- **影響するファイル数（見積もり）**: 10未満 / 10-30 / 30以上
- **実装工数（見積もり）**: 1-3日 / 3-7日 / 7日以上
- **各Agentのタスク数（見積もり）**: 5未満 / 5-10 / 10以上
- **複数の独立した機能を含むか**: はい / いいえ

#### 🚨 分割推奨の判定

以下の**いずれか1つでも該当**する場合、タスク分割を推奨します：

- ✅ 影響するファイル数が **30以上**
- ✅ 実装工数が **7日以上**
- ✅ 各Agentのタスク数が **10以上**
- ✅ API契約の破壊的変更 + DB変更が**同時発生**
- ✅ 複数の独立した機能を**同時実装**

**推奨される分割パターン:**

1. **フェーズ分割** - API/Backend → Frontend
   ```
   .steering/[YYYYMMDD]-[タスク名]-phase1-api/
   .steering/[YYYYMMDD]-[タスク名]-phase2-ui/
   ```

2. **サービス分割** - Backend → BFF → Frontend
   ```
   .steering/[YYYYMMDD]-[タスク名]-backend/
   .steering/[YYYYMMDD]-[タスク名]-bff/
   .steering/[YYYYMMDD]-[タスク名]-frontend/
   ```

3. **機能分割** - MVP単位で分解
   ```
   .steering/[YYYYMMDD]-[機能A]/
   .steering/[YYYYMMDD]-[機能B]/
   ```

**分割時の注意点:**
- 各ステアリングの`requirements.md`に前提となる他のステアリングを明記
- 依存関係を明確にし、実装順序を定義
- 共通のAPI契約は最初のステアリングで確定させる

### 3. 技術的変更点

#### 3.1 API契約
- **API変更の有無**: あり / なし
- **変更内容**: 新規エンドポイント追加 / 既存エンドポイント修正 / 削除
- **後方互換性**: 維持される / 破壊的変更

### 4. 非機能要件
- **パフォーマンス要件**: レスポンスタイム、スループット等
- **セキュリティ要件**: 認証・認可、データ保護等
- **J-SOX対応**: 監査ログ、承認フロー等

## ブランチ運用ルール

### 🚨 重要: ステアリングファイルはmainブランチで作成

ステアリングファイルの作成・設計はすべて **mainブランチ** で行います。
コードの実装は **featureブランチ** で行います。

**理由:**
- ステアリングファイルは「設計ドキュメント」であり、実装前に確定させるべきもの
- mainブランチに設計を置くことで、featureブランチ作成時に最新の設計を参照できる
- 設計と実装のライフサイクルを分離し、レビューしやすくする

**フロー:**
1. **mainブランチで** ステアリングファイル作成・レビュー・コミット・プッシュ
2. **featureブランチを作成** して実装開始（`/start-implementation` スキル使用）
3. 実装完了後、featureブランチからPR作成

**実行前チェック:**
- 現在のブランチが `main` であることを確認
- mainでない場合は `git checkout main && git pull origin main` を実行してから開始
- ステアリングファイル作成後は `git add .steering/ && git commit && git push origin main` でmainにプッシュ

## 実行フロー

1. **ブランチ確認**: mainブランチにいることを確認（mainでなければ切り替え）
2. **ヒアリング開始**: 上記の項目を順番に質問
3. **回答の整理**: 収集した情報を構造化
4. **タスクサイズ判定**: 分割が必要か判断し、必要な場合は分割方法を提案
5. **ステアリングディレクトリ作成**: `.steering/[YYYYMMDD]-[タスク名]/`
6. **ファイル生成**:
   - `requirements.md`: タスクの要求内容（依存する他のステアリングを明記）
   - `design.md`: 実装設計
   - `tasklist.md`: Agent別タスクリスト
7. **永続的ドキュメント更新提案**: `docs/`の更新が必要な場合は提案
   - `docs/glossary.md`: 新規ユビキタス言語の追加
   - `docs/service-contracts.md`: API契約の更新
   - その他関連ドキュメント
8. **mainにコミット・プッシュ**: ステアリングファイルをmainブランチにコミット・プッシュ

## 使用方法

```
/plan-task
```

このコマンドを実行すると、対話形式でヒアリングが開始されます。

---

## 注意事項

- すべての項目に回答する必要はありません（該当しない項目は「なし」で構いません）
- 不明な点があれば、後で調査・確認することも可能です
- ヒアリング終了後、ステアリングファイルのレビューを行います
